# UX Audit: SecuScan Frontend

**Date**: 2026-03-25
**Scenario**: "First-time user runs a scan and reviews output/findings"
**Browser**: Playwright (Chromium)
**Viewport**: Desktop (1280px) + Mobile (375px)

## Summary

The core desktop happy path is partially functional: a user can open Scanner, configure Network Scanner, start a task, and reach Task Details. However, several high-impact navigation defects create dead ends and route mismatches, and mobile users cannot discover the full app navigation needed to execute the same workflow.

## Findings

### Critical (blocks user task)

- **Scanner tabs appear selectable but lead to empty dead ends**: `Exploit Detection`, `Utils`, and `Robots` tabs render zero tool cards with no explanatory empty state.
  - *Where*: `/scans` tab navigation
  - *Screenshot*: `docs/ux-audit-artifacts/03-scanner-empty-exploit-detection.png`
  - *Fix*: Correct tab/category mapping in filtering logic (`exploit`, `utils`, `robots`) and add explicit empty-state guidance when a category is intentionally unavailable.

- **Mobile users cannot discover primary navigation to Scanner**: Sidebar is hidden below `lg`, and no global replacement menu is provided; visible links are contextual dashboard links and do not expose full route access.
  - *Where*: global shell on 375px (`/`)
  - *Screenshot*: `docs/ux-audit-artifacts/09-dashboard-mobile.png`
  - *Fix*: Add mobile global navigation (drawer/hamburger or bottom tabs) with direct links to core routes (`/scans`, `/findings`, `/reports`, `/history`, `/settings`).

### High (causes confusion or friction)

- **Tool config back button navigates to wrong route**: Back action sends users to `/scanner`, but configured app route is `/scans`.
  - *Where*: `/scans/:toolId` header back button
  - *Screenshot*: `docs/ux-audit-artifacts/05-broken-back-route.png`
  - *Fix*: Replace `/scanner` with `/scans` in ToolConfig navigation and add wildcard route fallback to prevent blank states.

- **Dashboard “Details” links navigate to non-existent route**: Clicking `Details` opens `/tasks`, which is not defined in the router.
  - *Where*: dashboard activity list `Details` CTA
  - *Screenshot*: `docs/ux-audit-artifacts/12-dashboard-details-broken-route.png`
  - *Fix*: Update CTA target to a valid route (`/history` or `/task/:taskId`) and align all task-link patterns across the app.

- **Theme controls in Settings are non-functional globally**: Choosing `Light` in `Visual Spectrum` does not alter app-level theme class/state.
  - *Where*: `/settings` theme selection controls
  - *Screenshot*: `docs/ux-audit-artifacts/11-settings-theme-no-effect.png`
  - *Fix*: Wire Settings controls to shared `ThemeContext` instead of local component state.

## What Works Well

- Desktop sidebar navigation is clear and consistently available.
- Scanner tool cards provide strong visual hierarchy and clear affordances on desktop.
- Network Scanner execution flow can successfully queue a task and navigate to `/task/:taskId`.

## Recommendations (priority order)

1. Fix route integrity and navigation consistency (`/scanner` vs `/scans`, `/tasks` links, fallback route handling).
2. Restore Scanner tab correctness and provide empty-state guidance for unavailable categories.
3. Implement mobile global navigation so core workflows are discoverable on 375px.
4. Connect Settings theme controls to `ThemeContext` so user preferences have observable effect.
