# Software Supply Chain Security & Release Integrity Guide

This document outlines the security controls, cryptographic signing practices, and verification steps used to maintain Software Supply Chain security and prevent release artifact tampering in SecuScan.

---

## 1. Supply Chain Threat Model & Release Risks

When deploying a security assessment framework like SecuScan, operators must trust that the code running on their servers has not been manipulated. Software supply chains face several critical threats:
*   **Asset Tampering**: Attackers hijacking the distribution channel (e.g. GitHub Releases) and replacing the official source or compiled tarballs with compromised versions containing backdoors.
*   **Man-in-the-Middle (MITM) Attacks**: Network-level attackers altering package assets during download transit.
*   **Build Hijacking**: Compromised developer accounts publishing malicious code under official release tags.

To mitigate these risks, SecuScan employs **automated cryptographic signing** for all official releases.

---

## 2. Cryptographic Signing with Sigstore (OIDC Keyless Flow)

Official SecuScan releases are signed using **Sigstore**, a Linux Foundation project designed to make software signing secure and transparent.

### How Keyless OIDC Signing Works
Rather than managing, storing, and risking the compromise of static private GPG/PGP keys, SecuScan leverages Sigstore's **Keyless OIDC (OpenID Connect) flow**:
1.  **OIDC Authentication**: During the official GitHub Release action, the runner requests a short-lived OIDC identity token from GitHub's OIDC Provider, certifying that the build is running inside the official repository's workflow (`.github/workflows/release.yml`).
2.  **Ephemeral Key Generation**: The workflow generates an ephemeral key pair on the runner machine.
3.  **Fulcio Certificate Authority**: The public key and OIDC token are sent to Fulcio (Sigstore's Certificate Authority). Fulcio issues a short-lived x509 certificate (valid for 10 minutes) binding the public key to the runner's GitHub workflow identity.
4.  **Signing**: The runner signs the release artifacts and the `SHA256SUMS` file with the ephemeral private key.
5.  **Rekor Transparency Log**: The signature and certificate are published to Rekor (Sigstore's public, immutable transparency ledger), allowing permanent, public auditing of who built and signed the release.
6.  **Destruction**: The ephemeral private key is immediately destroyed.

## 3. Step-by-Step Release Verification

To verify that your downloaded SecuScan release artifacts are authentic and untampered:

### Step 1: Download Release Assets
Download the following files from the official GitHub Release page into the same directory:
*   `secuscan-backend-<version>.tar.gz` (Backend source)
*   `secuscan-frontend-<version>.tar.gz` (Compiled frontend dist)
*   `SHA256SUMS` (Baseline checksums)
*   `SHA256SUMS.sig` (Cryptographic signature of the checksums)
*   `SHA256SUMS.certificate` (Fulcio certificate verifying build identity)

---

### Step 2: Validate Checksum Integrity
Run the built-in verification script included in the repository:
```bash
./scripts/verify-release.sh
```
This script automatically detects your OS and verifies that the downloaded archives match the hashes declared inside `SHA256SUMS`.

*If running manually:*
```bash
# On Linux:
sha256sum --check SHA256SUMS

# On macOS:
shasum -a 256 -c SHA256SUMS
```

---

### Step 3: Verify the Sigstore Signature
To cryptographically verify that the `SHA256SUMS` file was signed by the official SecuScan GitHub Actions workflow:

1.  **Install the Sigstore Client**:
    ```bash
    pip install sigstore
    ```
2.  **Run the Verification Command**:
    ```bash
    sigstore verify identity \
      --certificate SHA256SUMS.certificate \
      --signature SHA256SUMS.sig \
      --cert-identity "https://github.com/utksh1/SecuScan/.github/workflows/release.yml@refs/tags/v1.0.0" \
      --cert-oidc-issuer "https://token.actions.githubusercontent.com" \
      SHA256SUMS
    ```
    *(Be sure to replace `v1.0.0` with the actual release tag name you are auditing.)*

If verification is successful, Sigstore will confirm:
`OK: SHA256SUMS matches signature and certificate identity.`

---

### 3.1 Trusted OIDC Certificate Identity Reference

When verifying signatures generated by GitHub Actions, Sigstore validates that the signing certificate was issued to a trusted repository identity. For SecuScan, the trusted certificate identity parameters are defined as follows:

> [!IMPORTANT]
> **Official Upstream Releases (Tag-based)**:
> *   **Certificate Identity (Subject SAN)**: `https://github.com/utksh1/SecuScan/.github/workflows/release.yml@refs/tags/v*` (where `*` matches the specific release tag, e.g., `v1.0.0`)
> *   **OIDC Issuer**: `https://token.actions.githubusercontent.com`

> [!NOTE]
> **Manual Dry-run / Development Executions (Branch-based)**:
> *   **Certificate Identity (Subject SAN)**: `https://github.com/utksh1/SecuScan/.github/workflows/release.yml@refs/heads/<BRANCH_NAME>` (e.g., `.../release.yml@refs/heads/main` or `.../release.yml@refs/heads/release-provenance`)
> *   **OIDC Issuer**: `https://token.actions.githubusercontent.com`

Any certificate identity not matching `https://github.com/utksh1/SecuScan/` represents a fork or unofficial build and should not be trusted for official deployment.

## 4. Documented GPG Offline Signing Flow (Fallback)

For operators or developers packaging custom or offline distributions of SecuScan where GitHub OIDC is unavailable, a traditional GPG-based signing flow is supported.

### A. Generating a Release Key
```bash
# Generate a secure RSA 4096-bit key pair
gpg --full-generate-key
```

### B. Signing the Custom Release Checksums
1.  Assemble your custom tarball archives inside `dist/`.
2.  Generate the checksum baseline:
    ```bash
    cd dist/
    sha256sum secuscan-custom-backend.tar.gz secuscan-custom-frontend.tar.gz > SHA256SUMS
    ```
3.  Create a GPG clear-signed signature file:
    ```bash
    gpg --clearsign --digest-algo SHA256 -o SHA256SUMS.asc SHA256SUMS
    ```

### C. Verifying GPG-Signed Checksums
Operators downloading a custom GPG-signed release can verify the integrity using:

1.  **Import the Signer's Public Key**:
    ```bash
    gpg --import signer-public-key.asc
    ```
2.  **Verify the Checksum Signature**:
    ```bash
    gpg --verify SHA256SUMS.asc
    ```
3.  **Assert Checksum Alignment**:
    ```bash
    # Extracts the original checksum block and validates the files
    gpg --decrypt SHA256SUMS.asc | sha256sum --check
    ```
